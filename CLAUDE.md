Project context and conventions for working in this repo.

## What this project is

A **text-to-SQL evaluation lab** — an LLM agent that converts natural language questions into SQL queries, with a full evaluation harness to benchmark it. It is also exposed as a **FastAPI service** so the agent and eval suite can be consumed via HTTP.

The database under test is a toy e-commerce DuckDB instance (customers, products, orders, order_items) wrapped inside a 50-table enterprise data warehouse simulation (46 decoy tables) to stress-test schema linking.

## Running the project

### Server

```bash
uvicorn src.api.main:app --reload
```

Swagger UI at `http://localhost:8000/docs`.

### Evals (CLI)

```bash
python scripts/run_eval.py                          # default: gpt-4o-mini, zero_shot
python scripts/run_eval.py --strategy few_shot_dynamic
python scripts/run_eval.py --models openai/gpt-4o-mini anthropic/claude-haiku-4-5 --strategies zero_shot few_shot_dynamic
```

## Key design decisions

### Agent (`src/agent/agent.py`)

- All LLM calls go through **LiteLLM** — swap models by changing `DEFAULT_MODEL` env var, no code changes needed.
- **Self-correction loop** is implemented as a **LangGraph StateGraph**: `generate_node` → `execute_node` → conditional edge back to `generate_node` on SQL error (up to `max_retries=3`). This makes state transitions explicit and auditable.
- The `@observe` decorator from Langfuse traces every call automatically. `trace_id` is returned in the result and attached to eval metadata for cross-referencing.
- `generate_sql` is synchronous; FastAPI calls it via `asyncio.to_thread` to avoid blocking the event loop.

### Prompt strategies

| Strategy | How context is built |
|---|---|
| `zero_shot` | Schema + question only |
| `few_shot_static` | 3 hand-picked examples prepended |
| `few_shot_dynamic` | 3 examples selected by cosine similarity to the question embedding |
| `chain_of_thought` | System prompt instructs step-by-step reasoning; output parsed from `Reasoning: / SQL:` format |
| `rag` | Top-K table definitions retrieved by embedding similarity from 50-table DWH; schema not sent in full |
| `routed` | Classifies question difficulty (rule-based → embedding k-NN), then resolves to best model + strategy |
| `tool_use` | LLM chooses from `query_database`, `search_knowledge_base`, `get_schema` tools; handles SQL + policy + hybrid questions |

**Recommended strategy for SQL:** `few_shot_dynamic` — +40% result_match vs zero_shot.
**Recommended strategy for general assistant:** `tool_use` — handles policy and hybrid questions.

### RAG infrastructure (`src/rag/`)

- **Chunkers**: `FixedSizeChunker` (char + overlap), `SentenceChunker` (sentence boundaries), `SchemaChunker` (table-level)
- **Vector stores**: `InMemoryStore` (O(n) scan, no persistence) and `ChromaDBStore` (HNSW indexed, persistent). Both implement the same `VectorStore` abstract interface — swap backends without changing callers.
- **DocumentRetriever**: composes chunker + store; `index_file()` + `retrieve_text()` public API.
- `benchmark_rag.py` sweeps chunker × store × top-K and measures `recall@K`.
- Do not re-embed chunks on every call — the KB retriever in `tools.py` is a module-level singleton.

### Tool-use agent (`src/agent/tools.py`)

- `query_database(sql)` — executes SQL on DuckDB (read-only), returns rows as JSON or error
- `search_knowledge_base(query)` — RAG over `datasets/docs/ecommerce_kb.md`
- `get_schema(table_name?)` — schema introspection before writing SQL
- `AgentResult.tool_calls` logs every tool invocation for observability and eval
- `AgentResult.sql` is populated from the last `query_database` call so existing SQL scorers work with `tool_use`

### Evals (`src/evals/`)

- Harness: **Inspect AI**. Each run is a `Task` with a `Dataset`, `Solver`, and list of `Scorer`s.
- Scorers run in parallel: `syntax_valid` → `execution_ok` → `result_match` → `semantic_judge` → `avg_tool_calls`.
- Custom metrics: `avg_cost`, `avg_latency`, `avg_total_tokens`, `avg_attempts`, `avg_tool_calls`, `retrieval_recall`.
- `result_match` gives partial credit (0.5) when row count and column count match but values differ.
- `semantic_judge` uses **Instructor** with a `VerdictData` Pydantic model for type-safe structured outputs — replaces brittle `json.loads()` parsing.
- Do not change the `result_match` partial credit logic (0.5 for right shape) without updating the `semantic_judge` grounding rules — they are calibrated together.

### FastAPI service (`src/api/`)

- `POST /query` — wraps `generate_sql` with Pydantic request/response models.
- `POST /evals/run` — starts an Inspect AI eval in the background via `BackgroundTasks`, returns `job_id` (202 Accepted).
- `GET /evals/{job_id}` — polls job status and returns scores when complete.
- Job state is stored in an in-memory dict (fine for local use; swap for Redis in production).

### Model router (`src/agent/router.py`)

- Two-stage classifier: rule-based first (free), embedding k-NN fallback (one API call) when rule confidence < 0.30.
- Routing table: easy → zero_shot, medium → few_shot_dynamic, hard → few_shot_dynamic on HARD_MODEL.
- `HARD_MODEL` env var escalates hard questions to a stronger model. Defaults to `DEFAULT_MODEL` if not set.
- Golden embeddings are cached via `@lru_cache` — not re-embedded on every request.
- `routed` is a meta-strategy: it resolves to a real strategy before the LLM call. `AgentResult.routed_difficulty` and `router_method` record how the decision was made.

### Fine-tuning (`scripts/prepare_finetune_data.py` + `notebooks/finetune_llama.ipynb`)

- Data prep: combines golden (15) + synthetic tuning (40) → JSONL chat format, 90/10 train/eval split.
- Training: Llama 3.1 8B + LoRA (r=16) via Unsloth on Colab T4 GPU. ~15 minutes, ~$0 cost.
- Export: GGUF Q4_K_M for Ollama local serving. LiteLLM supports Ollama via `ollama/` model prefix.
- Eval: run `python scripts/run_eval.py --model ollama/text2sql-llama --strategy zero_shot` to benchmark.
- The hypothesis: a fine-tuned small model approaches few_shot_dynamic quality at zero marginal cost.

### DSPy optimizer (`scripts/optimize_prompt.py`)

- Uses `dspy.ChainOfThought(TextToSQL)` — the `TextToSQL` Signature declares `schema + question → sql`.
- `db_metric`: executes predicted SQL against DuckDB, compares normalized rows to expected — same logic as `result_match` scorer.
- `BootstrapFewShot` tries candidate demo combinations from `tuning.json`, keeps those that maximize `db_metric`.
- Compiled prompt saved to `datasets/dspy_optimized_prompt.json` — load it for inference without re-running optimization.
- Pre-fetches RAG schemas for each training example so demos match the inference-time prompt format exactly.

### Guardrails (`src/guardrails/`)

- **Input** (regex-based): blocks SQL injection patterns and prompt injection phrases before the LLM sees the question.
- **Output** (AST-based via sqlglot): enforces SELECT-only and schema allowlist after the LLM responds.
- **Execution** (DB-level): `execute_query` connects with `read_only=True` — even if a destructive statement bypasses the AST guardrails, DuckDB itself will reject it.
- Three-layer defense in depth: input regex → output AST → read-only connection.
- Known evasion gaps in input guardrails are documented as `xfail` in `tests/test_guardrails_adversarial.py`.

## What not to do

- Do not mock DuckDB in tests — use the real database. Mocks hide execution errors that only show up against the real engine.
- Do not re-embed table definitions on every call — the cache in `schema_retriever.py` exists for a reason.
- Do not change the `result_match` partial credit logic (0.5 for right shape) without updating the `semantic_judge` grounding rules too — they are calibrated together.
